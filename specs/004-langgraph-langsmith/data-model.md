# Data Model: LangGraph + LangSmith Integration

**Date**: 2026-03-08
**Status**: Complete

## Overview

This document defines the state schema, entities, and data structures for the LangGraph analysis workflow integrated into the Streamlit app.

---

## State Schema: `AnalysisState`

The complete LangGraph state is a TypedDict that persists across graph nodes via the checkpointer. Fields are intentionally designed to accommodate both Session 1 (happy path) and Session 2 (retry loop, cross-session persistence).

### AnalysisState TypedDict

```python
from typing import TypedDict, Optional
from datetime import datetime

class AnalysisState(TypedDict):
    """Complete state for analysis workflow."""

    # === Inputs ===
    user_query: str                    # User's natural language analysis request
    df_csv: str                         # Serialized CSV text of active DataFrame

    # === Plan Generation ===
    plan_steps: list[str]              # Ordered list of analysis steps (e.g., ["Step 1: Load data", "Step 2: ..."])
    plan_approved: Optional[bool]       # None = awaiting, True = approved, False = rejected
    plan_rationale: str                 # Optional explanation of plan (for logging/transparency)

    # === Code Execution ===
    code: str                           # Current generated or rewritten Python code
    execution_result: Optional[dict]    # Result from code execution: {"output_type": str, "content": Any, "error": Optional[str]}
    retry_count: int                    # Number of code rewrite attempts (0–3, reserved for Session 2)

    # === Report ===
    report_summary: str                 # Natural language summary of completed analysis
    execution_blocks: list[dict]        # List of {"step": str, "code": str, "result": dict} for Code tab

    # === Control ===
    cancelled: bool                     # Whether user cancelled the run (reserved for Session 2)
    error: Optional[str]                # Unrecoverable error message (session-level)

    # === Session 2 (Reserved) ===
    templates: list[dict]               # Session-scoped saved templates [{"name": str, "plan_steps": list, "code": str, "created_at": str}]
```

### Field Descriptions

#### Inputs
- **user_query**: Captured from chat input; passed to LLM for plan generation
- **df_csv**: CSV-serialized active DataFrame; included in plan generation context to make code specific to the data

#### Plan Generation
- **plan_steps**: List of numbered steps (e.g., `["1. Calculate correlation matrix", "2. Plot heatmap", "3. Summarize findings"]`). Generated by LLM, displayed in Plan tab.
- **plan_approved**: Session 1 auto-approves (True immediately); Session 2 uses interrupt/resume for manual approval
- **plan_rationale**: Optional transparency field; could log LLM reasoning (reserved for future use)

#### Code Execution
- **code**: Generated or rewritten code. Session 1 executes once; Session 2 retries (up to 3 rewrites) before fallback
- **execution_result**: Structured output from code execution: `{"output_type": "figure" | "dataframe" | "text" | "none", "content": Any, "error": Optional[str]}`
- **retry_count**: Tracks rewrite attempts. Session 1 always 0; Session 2 increments on failure (0–3 max)

#### Report
- **report_summary**: High-level summary written by LLM after successful execution (e.g., "Analysis complete: Found strong correlation between X and Y")
- **execution_blocks**: Ordered list of execution results for Code tab rendering. Each block: `{"step": str, "code": str, "result": {...execution_result...}}`

#### Control
- **cancelled**: Boolean. Session 1 always False; Session 2 set to True if user clicks Cancel
- **error**: Top-level error message if graph encounters an unrecoverable error (e.g., OpenAI API down). If set, graph halts and error is displayed.

#### Session 2 (Reserved)
- **templates**: List of saved templates. Session 1 uses JSON files; Session 2 may extend to database. Schema reserved here to avoid Session 1 → Session 2 migration issues.

## Entity: `ExecutionBlock`

Represents one code execution result in the Code tab report.

```python
class ExecutionBlock(TypedDict):
    """One code execution block for Code tab."""
    step: str                           # Description (e.g., "Step 1: Calculate correlation")
    code: str                           # Executed code (syntax-highlighted in UI)
    result: dict                        # execution_result: {"output_type", "content", "error"}
```

### Example

```python
{
    "step": "Step 1: Calculate correlation matrix",
    "code": "corr_matrix = df.corr()\nprint(corr_matrix)",
    "result": {
        "output_type": "text",
        "content": "           A    B    C\nA  1.000000  0.995....",
        "error": None
    }
}
```

## Entity: `AnalysisTemplate`

Represents a saved, reusable analysis plan.

```python
class AnalysisTemplate(TypedDict):
    """Saved template for reusable analysis."""
    name: str                           # User-provided template name (e.g., "Correlation Analysis")
    plan_steps: list[str]               # Copy of plan_steps when saved
    code: str                           # Copy of code when saved (optional; may be regenerated per data)
    created_at: str                     # ISO 8601 timestamp
```

### Storage
- **Session 1**: JSON files in `templates/` folder (one file per template, named by UUID + name)
- **Session 2**: May extend to database; schema slot reserved

### File Format
```json
{
    "name": "Correlation Analysis",
    "plan_steps": [
        "Step 1: Calculate correlation matrix",
        "Step 2: Generate heatmap visualization",
        "Step 3: Identify strongest correlations"
    ],
    "code": "import seaborn as sns\ncorr = df.corr()\nsns.heatmap(corr)...",
    "created_at": "2026-03-08T14:23:45.123456Z"
}
```

## Entity: `GraphRun`

Represents a single invocation of the analysis graph (not persisted; runtime only).

```python
class GraphRun:
    """Single graph execution context."""
    thread_id: str                      # Matches Streamlit session_id
    checkpointer: MemorySaver           # In-memory state storage
    state: AnalysisState                # Current state dict
    started_at: datetime                # Run start time
```

### Lifecycle
1. **Created**: When user submits analysis query
2. **Plan generated**: LLM generates plan_steps
3. **Auto-approved** (Session 1): plan_approved = True
4. **Code executed**: Python code runs in isolated namespace
5. **Report rendered**: execution_blocks populated
6. **Complete**: User views Code tab; run remains in session state for re-runs (cancel, retry in Session 2)

---

## Relationships

### AnalysisState → ExecutionBlock (1 → many)
- One analysis run generates multiple execution blocks (one per step)
- Blocks are accumulated in `execution_blocks` list

### AnalysisState → AnalysisTemplate (many → 1)
- Multiple runs may be based on the same template (load template, run with new data)
- Template stores `plan_steps` and `code`; state captures execution result

### GraphRun → AnalysisState (1 → 1)
- One graph run owns one state instance
- Checkpointer persists state across Streamlit reruns (same thread_id = same state)

## Validation Rules

### AnalysisState Invariants (Session 1)

1. **plan_steps.length > 0** if plan was generated
2. **plan_approved == True** (auto-approval in Session 1)
3. **code.length > 0** after plan generation (LLM always produces code)
4. **retry_count == 0** (no retry in Session 1)
5. **cancelled == False** (no cancel UI in Session 1)
6. **If execution_result.error is None**: execution_blocks should have one entry with result.content
7. **If execution_result.error is not None**: Report shows error; no execution_block added

### ExecutionBlock Invariants

1. **step**: Non-empty string
2. **code**: Non-empty string (or empty if step was conceptual, e.g., "Setup complete")
3. **result.output_type**: One of "figure", "dataframe", "text", "none"
4. **result.content**: Non-None if output_type != "none"

### AnalysisTemplate Invariants

1. **name**: Non-empty, unique within session
2. **plan_steps**: Non-empty list of strings
3. **code**: May be empty (code is regenerated per data in some use cases)
4. **created_at**: Valid ISO 8601 timestamp

## State Transitions (Session 1 Happy Path)

```
Initial → user_query + df_csv provided
  ↓
Generate plan → plan_steps populated, plan_approved = True
  ↓
Generate code → code populated
  ↓
Execute code → execution_result populated
  ↓
Render report → execution_blocks populated, report_summary generated
  ↓
Complete
```

---

## State Transitions (Session 2 Retry/Cancel, Reserved)

```
[From Complete above]
  ↓
If execution_result.error AND retry_count < 3:
  Rewrite code → code replaced, retry_count += 1
  ↓
  Execute code → new execution_result
  ↓
  [Loop back to "If execution_result.error"]

If execution_result.error AND retry_count == 3:
  Revise plan → plan_steps replaced, plan_approved = None
  ↓
  [User re-approves in UI] → plan_approved = True
  ↓
  Generate code → code replaced, retry_count = 0
  ↓
  Execute code

If user cancels:
  cancelled = True
  ↓
  [Graph exits]
```

## Summary

The `AnalysisState` TypedDict is the single source of truth for all workflow state. It accommodates both Session 1 (straight-line) and Session 2 (retry, cancel, persistence) without schema changes. Fields like `retry_count`, `cancelled`, and `templates` are reserved for Session 2 and initialized to default values in Session 1.
